# Идеи

## 1. Таблица сплитов на экране трека

На экране просмотра трека справа от карты показывать таблицу сплитов.

Логика расчета:

- для каждого КП искать точку трека с минимальным расстоянием до этого КП;
- искать точки последовательно по времени, чтобы порядок КП сохранялся;
- для каждого отрезка считать время между соседними найденными точками;
- показывать в таблице номер/метку КП, время прохождения, сплит и минимальную дистанцию.

Это даст быстрый автоматический разбор прохождения без ручной разметки кругов.

## 2. Опросник по проблемным сплитам

Для каждого проблемного сплита показывать короткий опросник, чтобы после разбора
сразу фиксировать причину потери.

Варианты ответа:

- я ошибся с направлением;
- выбрал плохой вариант пути;
- долго входил в КП;
- стоял и читал карту;
- бежал медленно, но без ошибки.

Логика:

- опросник показывать только для проблемных сплитов;
- по каждому проблемному сплиту нужно выбрать ответ;
- ответы сохранять вместе с разбором, чтобы потом видеть типичные причины потерь
  по дистанции и по серии стартов.

## 3. План тренировок на основе статистики проблем

Если по проблемным сплитам уже накоплены ответы из опросника, на их основе можно
строить сводную статистику по типам ошибок и дальше предлагать план тренировок.

Идея:

- агрегировать причины потерь по всем разобранным сплитам;
- показывать, какие проблемы встречаются чаще всего;
- по этой статистике формировать рекомендации по тренировочному фокусу.

Примеры:

- если часто встречается ошибка с направлением, нужен акцент на выходе в азимут и
  удержании направления;
- если часто встречается плохой выбор пути, нужен акцент на выбор варианта и
  чтение карты на скорости;
- если часто есть долгий вход в КП, нужен акцент на заходе в район пункта и
  взятии КП;
- если часто есть остановки на чтение карты, нужен акцент на чтении карты в
  движении;
- если потери чаще связаны с медленным бегом без ошибки, нужен акцент на беговой
  подготовке и темповых тренировках.

## 4. Несколько идеальных ориентиров в проблемной таблице протокола

В проблемной таблице протокола показывать не только строку `Идеальный лидер`, но
и дополнительные виртуальные строки:

- `Идеальное 2 место`;
- `Идеальное 3 место`.

Идея:

- собирать для каждого этапа не только лучший сплит, но и второй и третий по
  времени;
- строить виртуальных участников по этим наборам сплитов;
- показывать их рядом с текущим результатом в проблемной таблице.

Это позволит сравнивать себя не только с абсолютным идеалом по этапам, но и с
более реалистичными ориентирами уровня призовых мест.

## 5. Рисование варианта маршрута в анализе сплита

В диалоге анализа сплита добавить возможность вручную рисовать предполагаемый
вариант прохождения этапа на карте.

Идея:

- пользователь рисует альтернативный маршрут между КП;
- портал показывает рядом фактический трек прохождения и нарисованный вариант;
- можно визуально сравнить, где маршрут отличается;
- для нарисованного варианта можно оценивать длину и примерное время;
- дальше можно сравнивать фактический и нарисованный вариант и понимать, какой
  был лучше и на сколько.

Это сделает разбор выбора пути более предметным, а не только визуальным по
карте и треку.

## 6. Привязка карты к треку без спутниковой подложки

Добавить альтернативный режим геопривязки карты не к географической карте, а
напрямую к уже записанному треку.

Идея:

- на треке пользователь отмечает положения КП;
- на изображении карты пользователь отмечает те же КП;
- по этим соответствиям портал строит геопривязку карты;
- после этого карту можно использовать в плеере и анализе даже без спутниковой
  подложки.

Это позволит выполнять привязку в случаях, когда спутниковая карта неудобна,
неточна или вообще не нужна, а опорой для совмещения выступает сам трек и
известные контрольные пункты.
